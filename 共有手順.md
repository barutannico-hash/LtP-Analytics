# ダッシュボードを他チームと共有する手順（URL公開版）

ダッシュボードは `output/dashboard.html` の **1ファイルで完結**（データもグラフ設定も内包）しています。
これを静的ホスティングに置けば、**全員がURLを開くだけ**で閲覧できます。Python等のインストールは不要です。

> ドラフトシミュレーターは削除済みなので、**全チームに共有しても自チームの準備が漏れる心配はありません**。
> 共有されるのは各チームの戦績・チャンピオンプール等の分析結果です。

---

## 共有の全体像

1. **1人だけ**が `update.bat` を実行してデータを更新（`output/dashboard.html` が生成される）。
2. その `dashboard.html` をホスティングにアップロード → **URLが発行**される。
3. URLを他チームのメンバーに共有。各自ブラウザで開くだけ。
4. データを更新したら、**新しい `dashboard.html` を再アップロード**（同じURLのまま中身が更新）。

配るのは `dashboard.html` だけでOK。`data/` フォルダ（試合キャッシュ）はアップロード不要です。

---

## 方法A：Cloudflare Pages（おすすめ・無料・ドラッグ＆ドロップ）

無料・帯域無制限・HTTPS自動・アクセス制限もできて、ドラッグ＆ドロップで完結します。

1. アップロード用に **`dashboard.html` を `index.html` という名前にコピー**します
   （ホスティングはトップページに `index.html` を表示するため）。
2. [Cloudflare](https://dash.cloudflare.com/) に無料アカウントでログイン。
3. 左メニュー **「Workers & Pages」→「Create」→「Pages」→「Upload assets」** を選択。
4. プロジェクト名を入力（例 `evi-scrim-stats`）。
5. `index.html` を含むフォルダ（または `index.html` 単体）を**ドラッグ＆ドロップ**してデプロイ。
6. 発行された **`https://プロジェクト名.pages.dev`** が共有URLです。
7. 更新時は同じプロジェクトで「Create new deployment」から新しい `index.html` を再アップロード。

**アクセス制限したい場合**：Cloudflare の「Access（Zero Trust）」を使うと、指定したメールアドレスの人だけ閲覧可能にできます（少人数なら無料枠で運用可）。手軽に済ませるなら、推測されにくいプロジェクト名でURLを限定共有でもOKです。

---

## 方法B：Netlify Drop（アカウント不要に近い手軽さ）

1. `dashboard.html` を `index.html` にリネーム。
2. [https://app.netlify.com/drop](https://app.netlify.com/drop) を開く。
3. `index.html`（またはそれを含むフォルダ）を**ドラッグ＆ドロップ**。
4. 即座に `https://〇〇.netlify.app` のURLが発行されます。
5. 更新は同じ画面に再ドロップ、またはサイト設定から差し替え。

パスワード保護はNetlifyの有料機能です。無料で限定したい場合は方法A（Cloudflare Access）が便利です。

---

## 方法C：GitHub Pages（GitHubを使っている場合）

1. リポジトリを作成し、`index.html`（=リネームした dashboard.html）をプッシュ。
2. リポジトリの **Settings → Pages** で公開ブランチを設定。
3. `https://ユーザー名.github.io/リポジトリ名/` で公開。
4. 更新はファイルを上書きコミット。

※ 完全な非公開（限定公開）にはGitHub有料プランが必要です。社内限定にしたいなら方法Aが手軽です。

---

## 更新もWeb上で完結させる（GitHub Actions・ローカルPC不要）

「誰かがPCで `update.bat` を回す」のをやめ、**クラウド上で収集→集計→公開まで自動**にできます。
同梱の `.github/workflows/update.yml` がそれを行います（既存のPythonコードをそのまま実行）。

### セットアップ（最初の1回）
1. 無料の **GitHubアカウント** を用意。
2. **`lol_senseki_tool` フォルダの中身だけ**を新規リポジトリにアップロード（`.github/workflows/update.yml` も含める）。
   ⚠️ Cowork フォルダ全体を上げない。`data/`・`output/`・`players.csv` は **`.gitignore` で自動除外**されます
   （個人情報を含むため）。`config.json` の `api_key` はプレースホルダのままでOK。
3. **2つのSecretを登録**：リポジトリ **Settings → Secrets and variables → Actions → New repository secret**
   - `RIOT_API_KEY` … 自分のRiot APIキー（毎回手動で貼るなら省略可）。
   - `PLAYERS_CSV` … **選手リスト（サモナーネーム/Riot ID）の中身**。手元の `players.csv` をテキストエディタで開き、
     **中身を全部コピーして貼り付け**ます（ヘッダ行 `team,role,nickname,name,tagline` から全行）。
     これでRiot IDがリポジトリに残らず、実行時だけ復元されます。
4. **Settings → Pages → Source** を「**GitHub Actions**」に設定。

### 更新のやり方（Web上だけで完結）
1. リポジトリの **Actions タブ → 「Update dashboard」→ Run workflow**。
2. （Personalキーの場合）入力欄に**最新のAPIキーを貼って**実行。Secretを使うなら空のままでOK。
3. クラウドの仮想マシンが **収集→集計→ダッシュボード生成→GitHub Pagesへ公開**まで自動実行。
4. 完了後、deployジョブ（または Settings → Pages）に表示される
   `https://ユーザー名.github.io/リポジトリ名/` が共有URLです。

### 自動定期更新にしたい場合
`update.yml` の `schedule:`（cron）のコメントを外すと定期実行になります。
ただし **Personalキーは24時間で失効**するため、無人の定期実行は失効後に失敗します。
完全無人にするには Riot の **Productionキー**（失効なし・高レート）を申請してください。
手動「Run workflow」でキーを貼る運用なら、Personalキーのままでも**ローカルPC不要でWeb完結**します。

### 補足
- 初回はフル収集で時間がかかります（GitHub Actionsの1ジョブ上限は6時間。通常は収まります）。
  試合データはキャッシュされ、2回目以降は増分のみで高速です。
- 一時的に試合数を減らしたいときは Run workflow の入力欄 `match_count` で指定できます。
- GitHub Pages は無料だと公開URLです。**チーム限定にしたい**場合は、deployステップを
  Cloudflare Pages（Access）に差し替えると特定メールのみ閲覧に制限できます（必要なら手順を用意します）。

---

## 注意点

- **インターネット接続が必要**：グラフ（Chart.js）とチャンピオンアイコンはCDNから読み込みます。完全オフラインでは表は出ますがグラフ/画像が出ません（必要ならChart.js同梱版を別途用意できます）。
- **習熟度の手動編集は閲覧者ごとのブラウザに保存**されます。共有URLでも各自の編集は同期しません。チーム共通の習熟度にしたい場合は、エクスポート/インポート機能の追加で対応できます（要相談）。
- **更新は手動**：データを最新にするには、誰かが `update.bat` を実行して `dashboard.html` を再アップロードします。Windowsのタスクスケジューラ＋自動アップロードで定期更新を組むことも可能です。

---

## まとめ（最短ルート）

1. `update.bat` を実行 → `output/dashboard.html` を生成。
2. `index.html` にリネーム。
3. Cloudflare Pages に Upload assets でドラッグ＆ドロップ。
4. 発行された `*.pages.dev` のURLを全チームに共有。
